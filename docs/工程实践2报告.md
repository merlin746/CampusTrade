## 1 课题背景

### 1.1 课题背景

每到毕业季，校园里的跳蚤市场就热闹非凡——学长学姐们在宿舍楼下铺开摊子，出售用过的教材、电子产品、生活用品。但对于大多数同学来说，这种传统的地摊式交易有太多不便：你得专门腾出时间去摆摊或逛摊，信息靠贴在公告栏上的手写纸条传播，交易完成后连对方是谁都不知道，更别提售后维权了。而且一个学期也就那么一两周有集中的二手市场，平时想卖点闲置物品根本没有合适的地方。

另一方面，闲鱼这类大型二手平台虽然功能强大，但在校园场景下并不好用。首先，同学之间的交易往往局限在校内，互相邮寄既不划算也没必要，直接在宿舍楼下面交才最省事；其次，外面的人鱼龙混杂，同学间更容易建立信任，毕竟抬头不见低头见；再者，没有一个"校内专属"的标签，当全校几万人的闲置物品混在一个公共平台上时，找自己想要的东西就像大海捞针。

正是看到了这些痛点，我决定动手做这个"校园二手交易平台"。初衷很简单：让每一位同学都能随时随地把自己的闲置物品变成别人需要的宝贝，让"物尽其用"在校园里真正落地。

### 1.2 课题的目的

一句话概括，做这个平台就是为了**让同学之间买卖二手物品变得简单、安全、高效**。具体来说，有几个明确的目标：

第一，**把"摆摊"搬到线上**。不需要等到毕业季、不需要在楼下铺报纸，打开网页就能发布和浏览。想卖一个旧键盘？填一下信息、传两张照片，分分钟就上线了。

第二，**用审核机制建立信任**。所有发布的商品都要经过管理员审核，不是你想卖什么就挂什么。图片是否清晰、描述是否真实、价格是否合理，管理员看一眼心里有数。这就像给每一件商品都贴上了"已验真"的标签。

第三，**完整的交易闭环**。从浏览到下单，从卖家确认到买家收货，再到双方互评——整个过程都在平台上完成，每一步都有记录，谁也别想赖账。买卖双方的信誉分还会随着评价涨跌，这就倒逼大家诚信交易。

第四，**管理员能高效地"管好场子"**。仪表盘一看就知道今天新增了多少用户和商品，待审核的商品有多少件堆在那里，违规的可以立刻下架。不用像以前那样一个一个人工去查。

### 1.3 课题内容

这个平台面向三种角色：普通用户、管理员，还有其实也是普通用户但暂时以卖家身份出现的同学。每种角色看到和能做的事都不一样。

普通用户打开首页就能浏览所有在售商品，按分类筛选、按关键词搜索、按价格或时间排序——和在淘宝上逛没什么两样。点进商品详情，图片、描述、卖家信息、已有评价一清二楚。心动就点"立即购买"，跳到支付页面选微信或支付宝，扫码支付后订单就生成了。买家在"我的订单"里能随时查看进展——卖家确认了吗？可以收货了吗？全都一目了然。交易完成后双方互评，给自己攒信誉。不想买的可以先收藏，想卖的自己发布——填个表单传几张图，提交后等着管理员审核就行。

管理员用独立的暗色后台系统，一眼就能看到仪表盘上的核心数据。审核页面把每一件待审商品展示得清清楚楚——标题、图片、描述、卖家信息、分类、成色——是"通过"还是"不通过"或者"直接下架"，一键搞定。用户管理、商品管理、分类管理也都集成在后台。

为了让平台不显得空荡荡，我还提前插入了二十多件精心设计的示例商品——从高等数学教材到 iPad Air，从宿舍用小冰箱到永久牌山地车，覆盖了八个分类，每件商品都有三张精美的插图。这样打开首页就能看到一个琳琅满目的校园二手市场。

## 2 技术及功能需求

### 2.1 技术需求

这个项目的前后端是完全分离的，后端提供 API，前端用 Vue 搭界面，两者通过 HTTP 通信。具体的技术栈是：

| 层次 | 技术 | 版本 | 用途 |
|------|------|------|------|
| 后端框架 | Spring Boot | 3.4.3 | 控制器、业务逻辑、数据库操作 |
| 安全框架 | Spring Security + JWT | 6.4.3 / jjwt 0.12.6 | 用户认证、权限管理 |
| 数据库 | MySQL 8.0 + MyBatis-Plus 3.5.5 | — | 数据存储、ORM 操作 |
| 前端框架 | Vue 3 + Vite | 3.4 / 5.2 | 页面构建、热更新开发 |
| UI 库 | Element Plus | 2.7 | 表格、表单、弹窗、头像等组件 |
| 状态管理 | Pinia | 2.1 | 用户登录态管理 |
| HTTP 客户端 | Axios | 1.6 | 前端请求后端 API |
| API 文档 | Knife4j | 4.5 | 自动生成 Swagger 文档 |
| 工具库 | Hutool | 5.8.34 | 日期、加密、文件操作 |

选择这套技术的理由很实际。Spring Boot 搭起来的后端代码量不大但该有的都有——注解驱动的控制器、声明式事务、JWT 无状态认证——开发效率很高。MyBatis-Plus 在 MyBatis 的基础上又省了大量的 SQL 编写工作量，一个 `BaseMapper` 继承下来，常见的增删改查连一行 XML 都不用写。前端选 Vue 3 而不是 React，主要是因为 Vue 的模板语法对后端开发者更友好，学习曲线平缓，写起页面来像写 HTML 一样直观。Element Plus 提供了成熟的表格、表单、弹窗组件，外观精致且开箱即用，不用自己从零画组件。

### 2.2 功能需求

整个系统的功能按角色划分如下。实际给用户提供的功能远比这份列表更丰富，这里只列出核心部分。

**普通用户**

| 模块 | 功能点 |
|------|--------|
| 首页浏览 | 英雄区展示、分类筛选、关键词搜索、价格/时间/热度排序、分页浏览 |
| 商品详情 | 图片轮播、描述查看、评价列表、收藏/立即购买 |
| 支付页面 | 微信/支付宝扫码支付（演示模式） |
| 我的发布 | 查看已发布商品、编辑商品、下架商品、撤销发布（待审核状态）、删除历史记录 |
| 我的订单 | 查看买到/卖出的订单、确认收货、取消订单、评价对方、删除已完成/已取消订单 |
| 我的收藏 | 查看收藏商品、取消单件收藏、一键清空全部收藏 |
| 个人中心 | 修改个人资料、修改密码、查看收到的评价 |
| 商品评论 | 在商品页直接发表评论（无需下单）、匿名评论 |

**管理员**

| 模块 | 功能点 |
|------|--------|
| 仪表盘 | 总用户数、总商品数、总订单数、待审核数、今日新增数据（环形展示） |
| 用户管理 | 用户列表、搜索用户、启用/禁用用户 |
| 商品审核 | 待审核列表、审核通过/不通过/强制下架、拒绝原因填写 |
| 商品管理 | 全商品列表（按状态筛选）、强制下架操作 |
| 分类管理 | 分类列表、新增/编辑/删除分类 |

**用户 → 管理员交互流程**：用户发布商品 → 管理员审核通过 → 商品上架展示。如果审核不通过，卖家会收到系统通知并给出拒绝原因。

**买家 → 卖家交易流程**：买家浏览 → 下单 → 卖家确认 → 买家确认收货 → 双方互评。每一步都有对应的页面和按钮，状态流转清晰可控。

## 3 整体设计及需求

### 3.1 系统架构设计

整个系统采用前后端分离架构，前端是纯静态的 Vue 3 单页应用，通过 Vite 的开发服务器跑在 `localhost:5173`；后端是 Spring Boot 嵌入式 Tomcat，跑在 `localhost:8080`。Vite 配置了代理，所有 `/api` 和 `/uploads` 开头的请求都转发到 8080，开发阶段没有跨域问题。

```
浏览器 (localhost:5173)
    │
    ├── /api/*  ────────────────── Vite代理转发 ──→ Spring Boot (localhost:8080)
    │
    ├── /uploads/* ─────────────── Vite代理转发 ──→ 本地文件系统 (E:/schoolTrade/uploads)
    │
    └── 其余路径 ───────────────────→ Vite 静态服务 (Vue Router 接管)
```

后端的分层结构跟上学期学的标准 MVC 模式一致：

```
Controller 层
    │  接收请求、参数校验、调用 Service、返回统一 Result 对象
    ▼
Service 层（接口 + Impl 实现类）
    │  业务逻辑封装、事务管理、异常处理
    ▼
Mapper 层（MyBatis-Plus BaseMapper）
    │  数据库操作
    ▼
MySQL 数据库（8 张业务表）
```

额外的一些组件：
- **JWT 过滤器**（JwtAuthenticationFilter）在请求到达 Controller 之前就拦截了，从请求头解析 Token，把用户信息注入 Spring Security 上下文。
- **全局异常处理器**（GlobalExceptionHandler）兜底，任何没被 catch 的异常都会被转成规范的 `{ code:500, message:"xxx" }` 格式返回。
- **统一响应封装**（Result / PageVO / GoodsVO 等 VO 类）确保前端不管调哪个接口，拿到的 JSON 结构都长一个样。

### 3.2 页面设计

页面设计上，我没有直接用 Element Plus 的默认主题，而是从头定制了一套"豪华版"的设计系统。全局的 CSS 变量定义了从皇家蓝紫系渐变色到琥珀色、翡翠色的完整色板，所有的圆角都加大到 12-20px，阴影加了多层叠加让卡片更有立体感。导航栏是毛玻璃透明效果（`backdrop-filter: blur(24px)`），一滚动就有那种悬浮在半空中的高级感。

前端路由表如下：

| 路径 | 组件 | 说明 |
|------|------|------|
| `/` | Home.vue | 首页 —— 英雄区 + 商品搜索 + 商品网格 |
| `/login` | Login.vue | 登录 —— 毛玻璃卡片 + 渐变背景 |
| `/register` | Register.vue | 注册 —— 同上风格 |
| `/goods/:id` | GoodsDetail.vue | 商品详情 —— 图文 + 评论 + 直接评论表单 |
| `/pay/:goodsId` | Payment.vue | 支付页面 —— 微信/支付宝扫码演示 |
| `/publish` | Publish.vue | 发布/编辑商品 —— 图片上传 + 表单 |
| `/profile` | Profile.vue | 个人中心 —— 资料修改 + 密码 + 评价 |
| `/my-goods` | MyGoods.vue | 我的发布 —— 多种状态卡片 + 操作按钮 |
| `/my-orders` | MyOrders.vue | 我的订单 —— 双 Tab + 确认/收货/评价 |
| `/my-favorites` | MyFavorites.vue | 我的收藏 —— 商品网格 + 清空按钮 |
| `/admin/*` | 管理员页面组 | 仪表盘 / 用户管理 / 审核 / 商品管理 / 分类管理 |

所有页面都加了 `fade-in-up` 入场动画和交错延迟动画（staggered animation），卡片在悬停时有缩放上浮和光泽扫过（shine effect）的效果。

**管理员页面**用了暗色调的侧边栏（`#0f172a` → `#1e293b` 渐变），左侧 250px 宽，带活跃指示器（紫色竖条），右侧是白色顶栏+内容区。仪表盘的四张统计卡片分别用紫、琥珀、绿、红四种配色区别，数据后面还有半透明的大图标作为装饰。

## 4 详细设计

### 4.1 数据库设计

数据库一共建了 8 张表，表名全部是英文单数（这是 MyBatis-Plus 的约定的下划线转驼峰映射）：

```
school_trade 数据库
├── user             用户表         —— 用户名、密码(Bcrypt)、角色、状态、信誉分
├── category         商品分类表     —— 名称、图标(emoji)、排序
├── goods            商品表         —— 标题、描述、价格、分类、卖家、成色、状态
├── goods_image      商品图片表     —— 图片URL、排序
├── favorite         收藏表         —— 用户→商品的收藏关系（唯一约束）
├── orders           订单表         —— 订单号、买家、卖家、商品、价格、状态、评价标记
├── review           评价表         —— 订单关联、评分(1-5)、内容、匿名标记
└── notice           通知表         —— 接收用户、类型、标题、已读标记
```

**核心表之间的关系**：

```
user ──1:N── goods (发布)
user ──1:N── orders (作为买家或卖家)
user ──1:N── favorite (收藏)
goods ──1:N── goods_image (图片)
goods ──1:N── review (评价)
orders ──1:N── review (评价)
category ──1:N── goods (分类)
```

**商品状态枚举**是整个系统的核心状态机：

```
0 = 待审核  →  [管理员审核]  →  1 = 在售   →  [买家下单]  →  2 = 已售出
                               ↘  4 = 审核不通过
1 = 在售    →  [卖家下架/管理员强制下架]  →  3 = 已下架
```

**订单状态流转**：

```
0 = 待确认  →  [卖家确认]  →  1 = 已确认  →  [买家收货]  →  2 = 已完成
    ↓ 任意一方取消
3 = 已取消
```

这个设计覆盖了交易过程中所有可能的情况。特别值得一提的是 `buyerReviewed` 和 `sellerReviewed` 两个字段——它们分别标记买卖双方是否已经评价，这样就能确保"你评了别人、别人还没评你"的场景也被正常处理，同时防止重复评价。

### 4.2 接口设计

后端提供了 40+ 个 RESTful API，按模块分组如下：

| 模块 | 方法 | 路径 | 说明 |
|------|------|------|------|
| 认证 | POST | `/auth/register` | 注册（Bcrypt 加密存储密码） |
| | POST | `/auth/login` | 登录（返回 JWT Token，24 小时有效） |
| | GET | `/auth/me` | 获取当前用户信息 |
| 分类 | GET | `/category/list` | 获取全部分类（公开） |
| | POST/PUT/DELETE | `/category` | 增删改分类（管理员） |
| 商品 | GET | `/goods?keyword=&categoryId=&sortBy=0` | 搜索/筛选（公开，只显示在售） |
| | GET | `/goods/{id}` | 查询详情（自动增加浏览量） |
| | POST | `/goods` | 发布商品（待审核状态） |
| | PUT | `/goods/{id}` | 编辑商品（重置为待审核） |
| | PUT | `/goods/{id}/off` | 下架（仅卖家可操作） |
| | DELETE | `/goods/{id}/withdraw` | 撤销发布（仅待审核可删） |
| | DELETE | `/goods/{id}` | 删除历史记录（已下架/审核不通过） |
| | GET | `/goods/my` | 我的发布列表 |
| 收藏 | POST | `/favorite/{goodsId}` | 收藏 |
| | DELETE | `/favorite/{goodsId}` | 取消收藏（或从列表删除） |
| | GET | `/favorite/list` | 收藏列表 |
| | DELETE | `/favorite/clear` | 清空所有收藏 |
| 订单 | POST | `/order` | 创建订单（买家） |
| | PUT | `/order/{id}/confirm` | 卖家确认订单 |
| | PUT | `/order/{id}/cancel?reason=` | 取消订单 |
| | PUT | `/order/{id}/complete` | 确认收货，完成交易 |
| | GET | `/order/bought` | 我买到的 |
| | GET | `/order/sold` | 我卖出的 |
| | DELETE | `/order/{id}` | 删除历史记录（已完成/已取消） |
| 评价 | POST | `/review` | 创建评价（支持订单评价和直接商品评论） |
| | GET | `/review/goods/{goodsId}` | 获取商品评价列表 |
| 通知 | GET | `/notice/list` | 通知列表 |
| | PUT | `/notice/{id}/read` | 标记已读 |
| 管理员 | GET | `/admin/dashboard` | 仪表盘统计 |
| | GET | `/admin/users` | 用户列表 |
| | PUT | `/admin/users/{id}/toggle-status` | 启用/禁用用户 |
| | GET | `/admin/goods/pending` | 待审核商品列表 |
| | PUT | `/admin/goods/{id}/audit` | 审核（通过/不通过） |
| | PUT | `/admin/goods/{id}/off-shelf` | 强制下架 |
| | GET | `/admin/goods` | 全商品管理列表 |
| 文件 | POST | `/upload/image` | 上传图片（返回 URL） |

所有接口的请求和响应格式都是统一的：

```json
// 成功
{ "code": 200, "message": "success", "data": { ... } }

// 业务错误
{ "code": 400, "message": "用户名已存在", "data": null }

// 系统异常
{ "code": 500, "message": "服务器内部错误", "data": null }
```

分页接口统一返回 `PageVO`：

```json
{
  "code": 200,
  "data": {
    "records": [...],
    "total": 23,
    "current": 1,
    "size": 12,
    "pages": 2
  }
}
```

需要认证的接口在请求头里带上 `Authorization: Bearer <jwt_token>`。JWT 的载荷里存了 `userId`、`username` 和 `role`，后端通过 `JwtAuthenticationFilter` 解析后注入到 Spring Security 上下文中，Controller 通过 `@RequestAttribute("userId")` 就能拿到当前登录用户是谁。

## 5 核心需求及功能实现

### 5.1 前端页面展示

这个平台的前端页面全部由我亲手设计——没有用现成的模板，没有套什么 UI Kit。设计语言定调为"高端、精致、有活力"，整体走的是新拟态和玻璃态结合的风格。下面挑几个核心页面说说。

**首页 —— 第一眼就想逛下去**

打开首页，最先映入眼帘的是一块 520 像素高的"英雄区"（Hero Section）。背景是淡蓝紫色渐变 + 浮动模糊球体 + 网格图案叠加层，动态效果让整个区域仿佛"在呼吸"。正中央是大气的大标题和副标题，下面是一个圆角搜索框——不是默认的 `input`，而是用白色卡片包裹的自定义输入框，焦点时会有紫色边框和光晕。搜索框右边嵌了一个渐变色的搜索按钮。搜索框下面是四个热门搜索标签（🔥 教材、手机、耳机、自行车），点一下直接把关键词填进去。

往下滚动，首先看到的是分类胶囊栏——每个分类都是一个带 emoji 图标的小胶囊，选中时会变成紫色渐变填充。再往下是排序按钮组（最新 / 价格↑ / 价格↓ / 最热），像 iOS 的分段控件一样精致。最核心的商品网格每行自适应排列，每张商品卡片都做了丰富的细节处理：

- 图片区域悬停时会显示半透明黑色遮罩 + "查看详情"按钮
- 图片本身有 1.1 倍的缩放动画
- 左上角的"全新/几乎全新"成色标签是半透明毛玻璃效果
- 卡片悬停时会上浮 8px + 加阴影，还有一道从左到右扫过的光泽线条
- 价格用了大号红色粗体，如果有原价会显示删除线和折扣百分比标签
- 底部展示了卖家头像+昵称、浏览量和收藏量

**支付页面 —— 让人感觉真的在付钱**

支付页面的布局分左右两栏。左边是支付方式选择：微信和支付宝两个大卡片，每个卡片都有对应的绿色/蓝色图标和描述文字，点击选中后右侧会出现对勾标记，选中的卡片有紫色边框和光晕。右边上半部分是订单摘要——商品小图、名称、卖家、成色、价格，接着是实付金额（红色大字）。下半部分是核心亮点：一个 SVG 模拟的二维码，白色的底、黑色的定位图案和数据块随机分布，正中央叠了一个支付方式的 Logo 浮层。二维码下面写着"请使用微信扫一扫完成支付"和"扫码支付演示"的提示。

点击"确认支付 ¥XX"红色大按钮后，二维码区域被替换为一个动画——先是圆圈从无到有画出来（`stroke-dasharray` 动画），再是对勾描线动画，整个过程大概半秒，非常流畅。同时按钮显示 loading 转圈，2 秒后提示"支付成功"，自动创建订单并跳转。

**订单页面 —— 每一步都清清楚楚**

我的订单页面采用了一个简洁的 Tab 切换设计——🛒 我买到 / 💰 我卖出的。每张订单卡片从上到下分为三层：顶部是订单号和状态标签（待确认=橙、已确认=蓝、已完成=绿、已取消=红），中部是商品缩略图+标题+价格 + 对方的头像和名字，底部是操作按钮。买家能看到的按钮随订单状态变化：待确认时"取消"、已确认时"确认收货"、已完成未评价时"评价卖家"；卖家则是：待确认时"确认订单"、已完成未评价时"评价买家"。评价弹窗里有星级评分器、文字输入框、匿名评价复选框。

**管理员后台 —— 一眼掌握全局**

管理员的深色侧边栏设计非常有质感——品牌区有紫色渐变圆角图标盒子和白色字体，菜单项激活时有紫色竖条指示器和半透明紫色背景。仪表盘的统计卡片用了大号图标 + 大数字 + 背景水印图标的组合，悬停时浮起来。用户管理、商品管理都是现代化表格，状态用彩色标签标记，操作按钮内联在表格最后一列。

### 5.2 前端页面的设计

前端的设计系统是一套完整的 CSS 自定义属性体系。全局 `:root` 中定义了 40+ 个变量，从颜色到阴影到圆角到动画缓动函数，确保了整个项目的视觉一致性。

色彩系统的主色调选的是"皇室蓝紫"系（indigo → violet），因为这个色系既专业又不失活力，非常适合校园场景。强调色是琥珀橙，用于"立即购买"等关键行动按钮。语义色上，绿色=成功/在售、橙色=警告/待确认、红色=危险/下架/错误。

动画方面定义了 12 种关键帧动画：`fadeInUp`（淡入上浮）、`fadeInScale`（淡入缩放）、`float`（浮动）、`pulseGlow`（脉冲光晕）、`spin-slow`（慢速旋转）、`shimmer`（光泽扫过）、`gradientShift`（渐变流动）等等。页面上每个商品卡片、每条评价、每个统计数字都有自己的入场动画，通过 CSS 类 `stagger-1` 到 `stagger-12` 实现 12 级交错延迟（每级递增 50ms），让页面元素"依次出现"而不是"一下子全出来"。

### 5.3 后端的实现

后端代码一共约 4500 行，分布在 70 个 Java 文件中。总体架构遵循经典的 Controller → Service → Mapper 三层结构。

**认证与安全**是整个后端的基础设施。`SecurityConfig` 配置了无状态的 Spring Security，禁用了 CSRF（前后端分离不需要），指定了哪些路径公开访问、哪些需要登录、哪些必须是管理员。`JwtAuthenticationFilter` 在每个请求到达 Controller 前被调用，从 `Authorization` 头部提取 Bearer Token，用 `JwtUtil` 解析出 userId、username 和 role，构造 `UsernamePasswordAuthenticationToken` 并存入 `SecurityContextHolder`。`JwtUtil` 使用了 HMAC-SHA384 算法签名，密钥从 `application.yml` 注入，Token 有效期为 24 小时。

**商品模块**是业务的核心。`GoodsServiceImpl.publish()` 方法在保存商品时自动设置 `status=0`（待审核）、`viewCount=0`、`favoriteCount=0`，然后遍历 `dto.getImages()` 批量插入 `goods_image` 表。搜索方法使用 MyBatis-Plus 的 `LambdaQueryWrapper` 动态构建查询条件——关键字用 `like`、分类用 `eq`、价格区间用 `ge/le`、排序用 `orderByAsc/Desc`，每个条件都是按需拼接的，不会出现"空 where"问题。`convertToVO()` 方法负责把实体转换成前端需要的 VO，顺便把分类名称、卖家名称、图片列表都查出来填进去。

**订单模块**的状态流转逻辑非常严谨。创建订单时做了三重校验：商品是否存在、商品是否在售、是否买自己的东西。下单后商品状态直接变为"已售出"（status=2），防止一人下单后另一个人又买。卖家确认时校验卖家身份和订单状态（必须是 status=0）。买家收货时校验买家身份和订单状态（必须是 status=1）。取消时校验操作人是否是买家或卖家，然后把商品状态恢复到"在售"（status=1），这样商品就回到货架上了。

**评价模块**支持两种模式：有 `orderId` 的就是订单评价（买卖双方互评），有 `goodsId` 但没有 `orderId` 的就是直接商品评论（围观群众评价）。订单评价会触发订单评价标记的更新，防止重复评价；同时根据评分更新被评价人的信誉分——每比基准分 3 分高 1 分，信誉分 +5，低 1 分则 -5。直接商品评论只需传 goodsId、评分和内容，不需要先下单。两种评价都会更新商品的 `avg_score` 和 `review_count`。

**文件上传**用的是 Spring 标准的 `MultipartFile` 接口，保存路径配置在 `application.yml` 的 `file.upload-path` 中。由于 Spring Boot 的工作目录默认在临时目录，我把上传路径改成了绝对路径 `E:/schoolTrade/uploads`，避免文件找不到的问题。同时配置了 `WebMvcConfig` 的 `addResourceHandlers`，把 `/uploads/**` 映射到本地文件系统，图片就能通过 HTTP 直接访问了。

## 6 设计中存在的问题

做这个项目的过程中，确实踩了不少坑，也发现了一些设计上可以改进的地方。下面列出几个在开发过程中遇到并已经解决了的问题。

**YAML 密码前导零的坑**：项目最初在 `application.yml` 中设置了数据库密码 `password: 060423`，MySQL 命令行能正常连接，但 Spring Boot 连的时候报 `Access denied`。原因是 YAML 解析器把 `060423` 当成了数字（八进制），去掉了前导零。解决方案是加引号变成字符串 `password: "060423"`。这真的是一个经典又隐蔽的坑，花了大概半个小时才定位到。

**文件上传路径的坑**：最初配置的是 `upload-path: ./uploads`，心想用相对路径多方便。结果上传图片的时候报 `FileNotFoundException`，路径指向了 `C:\Users\Administrator\AppData\Local\Temp\tomcat.xxx\work\...\uploads`——这是 Tomcat 的工作目录，根本不是项目的根目录。改成绝对路径 `E:/schoolTrade/uploads` 后问题解决。这提醒我，在 Spring Boot 中任何涉及文件路径的配置都尽量用绝对路径，或者在启动时动态获取项目根目录。

**Element Plus Upload 组件和 fileList 不同步**：发布商品页的图片上传功能最初有个 bug——用户明明上传了图片，但提交时后端收到的 `images` 数组是空的。排查发现是 `el-upload` 组件内部维护的文件列表和 Vue 的 `ref` 没有正确同步。`handleUploadSuccess` 回调里虽然设置了 `file.url`，但提交时从 `fileList.value` 里取的 URL 对不上。解决方案是引入一个独立的 `uploadedImages` 数组，上传成功时直接 `push(response.data)`，删除图片时从数组里移除，提交时直接拿这个数组。这样绕过了 Element Plus 内部的状态管理，问题根治。

**多标签页串号问题**：这是一个很隐蔽的 bug。浏览器在同时打开多个标签页、用不同账号登录时，标签 B 的登录操作会覆盖 `localStorage` 中的 token，导致标签 A 下一次发请求时带上了标签 B 的 token，身份就串了——页面显示的是 A 的用户名，但请求的却变成了 B 的权限。解决方案是"三层防御"：第一层，`request.js` 拦截器不再从 `localStorage` 读 token，而是从 Pinia 内存中读取（每个标签页有独立的 Pinia 实例）；第二层，`Publish.vue` 的上传 header 也改用 Pinia token；第三层，`stores/user.js` 新增了 `safeRestore()` 方法，页面初始化时对比 token 中的 userId 和 user 对象中的 userId，如果不一致（说明被其他标签污染了）就自动清除。

**前端路由命名不一致**：右上角下拉菜单的 `command` 值用了驼峰命名（`myGoods`），但路由定义是短横线（`my-goods`），`handleMenu` 函数里直接用 `router.push('/${command}')` 就把 `myGoods` 拼成了 `/myGoods`，而路由表里根本没有 `/myGoods` 这条路径，结果是点了没反应。花了好一阵才意识到问题出在那几个下拉菜单的命令值上，改成短横线后一切正常。

除了已经修复的 bug，还有一个值得反思的设计问题：**评论系统最初设计得过于严格**——只有完成订单后才能评价。这样做的好处是防止刷好评，但也让很多潜在的好评无法产生。后来我加上了"直接商品评论"的功能，把 `ReviewDTO` 加了一个可选的 `goodsId` 字段，当没有 `orderId` 时就走直接评论逻辑，这样即使没下过单的人也能分享自己的看法。这个改动让评论区的活跃度明显提升了。

## 7 工程情况

这个项目从头到尾是我一个人独立设计和开发的，历时大约**两周半**，累计编写后端代码约 **4500 行**（70 个 Java 文件），前端代码约 **3800 行**（16 个 Vue 组件 + 8 个 JS 模块 + 全局样式），数据库表 **8 张**，API 接口 **40 余个**，还生成了 23 件示例商品的 69 张 Illustrator 风格插图。

整个开发过程经历了几个阶段：

**第一周**：搭好 Spring Boot 脚手架的骨架，写完了数据库初始化脚本，把 8 张表都建好，插入了默认的分类和管理员账号。然后用 MyBatis-Plus Generator 生成了基本的 Mapper 和 Entity，手写了 Controller 和 Service 的核心逻辑——认证、商品 CRUD、订单流转、评价计算。这个阶段每天平均写 500-800 行代码。

**第二周**：重点转向前端。第一天把 Vue Router 的路由树搭好，Pinia 的登录状态管理写好，导航栏和页脚做出来。之后每天攻克 3-4 个页面：首页、商品详情、登录注册、我的发布、我的订单、个人中心、管理后台。每个页面除了实现功能，还花大量时间打磨视觉效果——配色、动画、阴影、圆角、过渡效果。有几天晚上改 CSS 改到凌晨一两点。

**第三周的前半周**：查缺补漏。图片上传修复、多标签串号修复、撤销发布功能补完、支付页面实现、清空历史记录功能、直接商品评论功能。同时生成示例商品图片并插入数据库，让平台打开就有内容可看。

**生产环境搭建**：后端用 Gradle 构建，`./gradlew bootRun` 一键启动。前端用 `npm run dev` 启动 Vite 开发服务器。数据库是本地 MySQL 8.0，用 `init.sql` 一次性完成建库建表。详细的启动步骤已经总结在 `启动指南.md` 中。

## 结论

这次工程实践让我真正从头到尾走了一��前后端分离项目的完整开发流程。从技术层面来说，之前课堂上学的 Spring Boot、MyBatis、Vue 都是在做 demo 级别的玩具项目，而这次是一个真正五脏俱全的完整应用——8 张表、40+ 个接口、16 个页面、完整的用户认证和权限体系、复杂的商品和订单状态流转逻辑。这种复杂度下的开发体验和做一个 "Hello World" 是完全不一样的。

尤其是调试那些看起来"不可能发生"的 bug 的过程——YAML 数字前导零、Tomcat 临时目录、Element Plus 状态不同步、多标签 localStorage 串号——每一个都很折磨人，但每一个解决后都让我的理解深了一层。现在我终于明白为什么很多经验文档里都强调"认证状态不能只靠 localStorage"、"Spring Boot 的 file.upload-path 要配绝对路径"——因为书上写的一行字，往往对应着真实开发中一个小时的排查时间。

另外我也体会到，**前端不仅仅是"把界面画出来"**。一个设计精美的页面需要大量的 CSS 细节：配色方案的一致性、字体的层级关系、阴影的层次感、动画的缓动函数、悬停和点击的反馈——这些细节单独看都不起眼，但加在一起就是"好看"和"普通"的分水岭。我在这套设计系统上花的时间也证明了这一点是值得的——最后打开浏览器看到首页英雄区和商品卡片交互效果的时候，确实有一种"这就是我想要的东西"的感觉。

当然，这个平台也还有很多可以继续完善的地方：目前还没有用户间即时通信功能，所有沟通都靠订单状态和评价；还没有给商品加上"议价"功能，所有交易都是固定价格；还没有接入真实的微信支付和支付宝支付；搜索目前只是模糊匹配标题，还可以做得更智能。如果有机会继续迭代，这些都是很好的方向。

总的来说，这个项目的收获超出了我的预期。它不仅让我实践了前后端开发的技术栈，更让我从一个产品的角度去思考"什么是用户真正需要的功能"、"什么样的交互能让用户感到愉悦"、"什么样的代码组织能让后续维护不痛苦"。这些经验，我想在以后的工作中会有更多的用武之地。
